# BUG-001 — Vazamento de memória no `inventory-service`

> **Aviso**: este documento contém a causa raiz e a solução completa. Só leia
> se você já tentou investigar sozinho e está travado, ou quer conferir seu
> diagnóstico depois de concluir a investigação.

## Sintoma observável

Sob carga sustentada, o pod do `inventory-service` apresenta uso de heap
crescente e monotônico (não cai após GC), até que:

- o consumo de memória do processo se aproxima do `limits.memory` (512Mi)
  definido em `k8s/inventory-service.yaml`;
- o pod eventualmente é `OOMKilled` pelo kubelet, ou o processo lança
  `java.lang.OutOfMemoryError: Java heap space`;
- as `readiness`/`liveness probes` (`/actuator/health/*`) começam a falhar
  por lentidão (GC overhead), causando reinícios do pod (`CrashLoopBackOff`
  intermitente) e picos de latência em `order-service` e `shipping-service`,
  que dependem de `inventory-service`.

Nada nos logs de aplicação indica exceções óbvias — o serviço continua
respondendo normalmente às requisições até o momento em que a memória se
esgota.

## Causa raiz

Arquivo: `inventory-service/src/main/java/com/example/inventory/SupplierStockCache.java`

```java
@Component
public class SupplierStockCache {

    private final Map<String, Map<String, Object>> cache = new ConcurrentHashMap<>();

    public Map<String, Object> getOrFetch(String sku, String correlationId, Supplier<Map<String, Object>> fetcher) {
        String cacheKey = sku + ":" + correlationId;
        return cache.computeIfAbsent(cacheKey, key -> fetcher.get());
    }

    public int size() {
        return cache.size();
    }
}
```

Usado em `InventoryController.getItem(String sku)`:

```java
String correlationId = currentTraceId();
Map<String, Object> supplierStock = supplierStockCache.getOrFetch(
        sku, correlationId, () -> supplierClient.getSupplierStock(sku));
```

Esta classe foi introduzida como uma primeira tentativa de atacar o item
"Cache" do roadmap (`HELP.md`), com o objetivo de evitar chamadas repetidas
ao `supplier-service` para o mesmo SKU. Ela tem **dois problemas
independentes**, mas o segundo é o que causa o vazamento:

1. **Chave de cache incorreta** — a chave é `sku + ":" + correlationId`,
   onde `correlationId` é o trace ID atual (`Tracer.currentSpan()`), ou um
   `UUID` aleatório quando não há span ativo. Como o trace ID é único por
   requisição, **a chave nunca se repete** — o cache tem taxa de acerto
   (`hit rate`) igual a zero. Ele nunca cumpre sua função original de
   cache.

2. **Nunca há remoção de entradas** — como cada chave é única, todo
   `computeIfAbsent` insere uma entrada nova em vez de reaproveitar uma
   existente. Não há TTL, não há tamanho máximo (`maximumSize`), não há
   política de evicção (LRU/LFU), e nada jamais chama `cache.remove(...)`
   ou `cache.clear()`. O `Map` é mantido como campo de um `@Component`
   `singleton`, ou seja, vive por todo o ciclo de vida da aplicação.

O resultado é um `ConcurrentHashMap` que cresce **indefinidamente**, uma
entrada por requisição a `GET /inventory/{sku}`, retendo referências fortes
a objetos (`Map<String, Object>` aninhados) que nunca são coletados pelo
GC. Isso é um vazamento de memória clássico por **cache sem limite**
("unbounded cache"), um dos padrões mais comuns de memory leak em
aplicações Java de produção.

## Como isso se manifesta na prática

- 1 requisição a `GET /inventory/{sku}` → 1 entrada nova no mapa (nunca uma
  reutilização), independentemente de quantas vezes o mesmo SKU for
  consultado.
- Sob um teste de carga sustentado (dezenas de milhares de requisições),
  o heap do `inventory-service` cresce de forma linear com o número de
  requisições, visível em Grafana/Prometheus através das métricas
  `jvm.memory.used{area="heap"}` expostas por `/actuator/prometheus`.
- Como o heap padrão do container é calculado a partir do `limits.memory:
  512Mi` (sem `-Xmx` explícito no `Dockerfile`), o heap máximo efetivo é
  relativamente pequeno (uma fração desse limite), então o OOM pode ser
  reproduzido em minutos com carga moderada (ex.: algumas dezenas de
  requisições por segundo).

## Como confirmar o diagnóstico

1. Observar em Grafana/Prometheus o painel de heap do `inventory-service`
   (métrica `jvm_memory_used_bytes{application="inventory-service",area="heap"}`)
   durante um teste de carga contra `GET /inventory/{sku}` — a curva sobe e
   nunca recua após os ciclos de GC (ao contrário de um vazamento comum que
   ao menos oscila).
2. Tirar um heap dump do pod (`jmap -dump:live,format=b,file=/tmp/heap.hprof
   <pid>` via `kubectl exec`, ou habilitar `management.endpoint.heapdump`
   e baixar via `/actuator/heapdump`) e analisar com Eclipse MAT — o
   "dominator tree" vai apontar `SupplierStockCache.cache` como o maior
   consumidor de heap, com um número de entradas muito próximo do número
   total de requisições recebidas.
3. Alternativamente, adicionar um log temporário ou endpoint de debug que
   exponha `supplierStockCache.size()` e observar que ele cresce 1:1 com o
   número de chamadas a `GET /inventory/{sku}`, nunca estabilizando — o
   que já comprova que o cache não está fazendo cache.

## Solução

> **Atualização**: a correção aplicada neste repositório usa Redis como
> cache distribuído (in-cluster), não Caffeine — decisão registrada em
> [`docs/ADR/0001-cache-de-estoque-redis.md`](ADR/0001-cache-de-estoque-redis.md),
> motivada pelo HPA do `inventory-service` (cache local não é compartilhado
> entre réplicas). As seções abaixo descrevem o raciocínio original com
> Caffeine; os dois problemas de fundo (chave errada e ausência de
> limite/TTL) são os mesmos e foram corrigidos de forma análoga com Redis.

Duas correções, uma para cada problema:

1. **Corrigir a chave do cache** — remover o `correlationId` da chave. A
   chave deve ser apenas o `sku`, que é o dado que de fato queremos
   cachear (o estoque do fornecedor para aquele SKU):

   ```java
   public Map<String, Object> getOrFetch(String sku, Supplier<Map<String, Object>> fetcher) {
       return cache.computeIfAbsent(sku, key -> fetcher.get());
   }
   ```

   Isso já limita naturalmente o tamanho do cache à cardinalidade de SKUs
   existentes, mas **não é suficiente por si só** em um catálogo real (com
   milhares/milhões de SKUs, ou SKUs que mudam de valor ao longo do tempo o
   cache voltaria a crescer sem limite ou a servir dados obsoletos).

2. **Adicionar limite de tamanho e expiração (TTL)** — usar uma biblioteca
   de cache com política de evicção, em vez de um `Map` cru. Recomenda-se
   [Caffeine](https://github.com/ben-manes/caffeine):

   ```xml
   <dependency>
       <groupId>com.github.ben-manes.caffeine</groupId>
       <artifactId>caffeine</artifactId>
   </dependency>
   ```

   ```java
   @Component
   public class SupplierStockCache {

       private final Cache<String, Map<String, Object>> cache = Caffeine.newBuilder()
               .maximumSize(1_000)
               .expireAfterWrite(Duration.ofMinutes(5))
               .build();

       public Map<String, Object> getOrFetch(String sku, Supplier<Map<String, Object>> fetcher) {
           return cache.get(sku, key -> fetcher.get());
       }

       public long size() {
           return cache.estimatedSize();
       }
   }
   ```

   Isso garante um teto de memória previsível (`maximumSize`) e dados que
   não ficam obsoletos indefinidamente (`expireAfterWrite`), mesmo que o
   número de SKUs cresça sem limite no futuro.

### Correções complementares (defesa em profundidade)

- Expor `cache.estimatedSize()` como métrica Micrometer
  (`Caffeine.recordStats()` + `CaffeineCacheMetrics.monitor(...)`) para que
  o tamanho do cache fique visível em Grafana e alertas possam ser
  configurados antes de um OOM acontecer novamente.
- Considerar definir explicitamente `-XX:MaxRAMPercentage` (ou `-Xmx`) no
  `Dockerfile`, alinhado ao `limits.memory` do Kubernetes, para tornar o
  comportamento de memória do JVM mais previsível e os OOMs mais rápidos de
  reproduzir/diagnosticar em testes.
- Adicionar um teste de carga automatizado (ex.: k6 ou Gatling) no CI que
  rode um número alto de requisições contra `GET /inventory/{sku}` e falhe
  se `jvm_memory_used_bytes` crescer de forma monotônica sem estabilizar —
  isso pegaria essa classe de bug antes de chegar a produção.

## Arquivos alterados na correção final (Redis)

- `inventory-service/src/main/java/com/example/inventory/SupplierStockCache.java`
  (chave agora é só o `sku`, com TTL e leitura/escrita fail-open no Redis)
- `inventory-service/src/main/java/com/example/inventory/InventoryController.java`
  (removido o uso de `correlationId`/`Tracer` na chamada ao cache)
- `inventory-service/src/main/java/com/example/inventory/RedisConfig.java` (novo)
- `inventory-service/pom.xml` (dependência `spring-boot-starter-data-redis`)
- `inventory-service/src/main/resources/application.yml` (config de conexão e TTL)
- `k8s/redis.yaml` (novo — Deployment + Service do Redis in-cluster)
- `k8s/inventory-service.yaml` (env `REDIS_HOST`/`REDIS_PORT`)
- `docs/ADR/0001-cache-de-estoque-redis.md` (novo — decisão arquitetural)
