# BUG-002 — Vazamento de threads no `notification-service`

> **Aviso**: este documento contém a causa raiz e a solução completa. Só leia
> se você já tentou investigar sozinho e está travado, ou quer conferir seu
> diagnóstico depois de concluir a investigação.

## Sintoma observável

Sob carga sustentada contra `POST /notifications`, o pod do
`notification-service` apresenta um comportamento que parece, a princípio,
um problema de CPU ou de rede, mas que na verdade é de esgotamento de
recursos do processo:

- a contagem de threads vivas da JVM cresce de forma monotônica (não cai
  nunca), visível via `jvm.threads.live` exposto em `/actuator/prometheus`;
- depois de tempo suficiente sob carga, novas requisições passam a falhar
  com stack traces contendo `java.lang.OutOfMemoryError: unable to create
  native thread` (às vezes precedido de erros do SO como `Resource
  temporarily unavailable` ao criar a thread);
- o pod pode ficar visivelmente mais lento antes de travar (troca de
  contexto entre milhares de threads a maior parte ociosas), e as
  `liveness`/`readiness probes` (`/actuator/health/*`) começam a falhar,
  levando a reinícios (`CrashLoopBackOff` intermitente);
- assim como no BUG-001, isso propaga latência e erros para os serviços que
  dependem de `notification-service` (chamado a partir do fluxo de
  pagamento/envio).

Diferente do BUG-001, aqui o heap não é o recurso que se esgota — o problema
é o número de threads do processo (memória nativa/stack por thread e limite
de threads do sistema operacional/`cgroup`).

## Causa raiz

Arquivo: `notification-service/src/main/java/com/example/notification/NotificationController.java`

```java
@PostMapping
public ResponseEntity<Map<String, Object>> createNotification(@RequestBody Map<String, Object> request) {
    String paymentId = (String) request.getOrDefault("paymentId", "UNKNOWN");
    String sku       = (String) request.getOrDefault("sku", "UNKNOWN");

    // Busca pagamento e envio em paralelo para reduzir a latência da notificação.
    ExecutorService executor = Executors.newFixedThreadPool(2);
    Future<Map<String, Object>> paymentFuture  = executor.submit(() -> paymentClient.getPayment(paymentId));
    Future<Map<String, Object>> shippingFuture = executor.submit(() -> shippingClient.getShipment(sku));

    Map<String, Object> paymentInfo;
    Map<String, Object> shippingInfo;
    try {
        paymentInfo  = paymentFuture.get();
        shippingInfo = shippingFuture.get();
    } catch (InterruptedException | ExecutionException e) {
        throw new RuntimeException(e);
    }

    return ResponseEntity.status(201).body(Map.of(
            "notification", request,
            "paymentStatus", paymentInfo,
            "shippingStatus", shippingInfo
    ));
}
```

Essa mudança foi introduzida como uma otimização aparentemente razoável:
buscar o status do pagamento e do envio **em paralelo**, em vez de
sequencialmente, para reduzir a latência de `POST /notifications` (as duas
chamadas HTTP, a `payment-service` e a `shipping-service`, não dependem uma
da outra).

O problema é onde e como o `ExecutorService` é criado:

1. **Um `ExecutorService` novo é criado a cada requisição**, dentro do
   método do controller, em vez de ser um recurso compartilhado (bean
   singleton) criado uma única vez na inicialização da aplicação.

2. **Esse `ExecutorService` nunca é encerrado** — não há `executor.shutdown()`
   nem `executor.close()` (try-with-resources), nem em um bloco `finally`,
   nem em nenhum outro lugar do código. `Executors.newFixedThreadPool(2)`
   cria um pool cujas *core threads* **não** têm tempo de vida limitado por
   padrão — elas ficam vivas indefinidamente esperando por novas tarefas,
   até que o pool seja explicitamente encerrado.

O resultado: **cada requisição a `POST /notifications` cria 2 threads
novas da plataforma que nunca são destruídas**. As threads anteriores nunca
são reaproveitadas (cada `ExecutorService` é descartado depois do uso, mas
suas threads continuam vivas e o objeto pool, sem mais referências,
eventualmente é apenas o pool que é coletado pelo GC — as **threads que ele
criou não são**, pois threads são recursos do sistema operacional geridos
pela JVM, não objetos comuns do heap). Isso é um vazamento de threads
("thread leak") clássico por **pool de execução de vida curta sem
`shutdown()`**, um dos padrões mais comuns de esgotamento de recursos em
aplicações Java que usam `ExecutorService` manualmente.

Cada thread da JVM consome memória nativa para sua stack (`-Xss`, por
padrão 512KB–1MB dependendo da plataforma) fora do heap, além de uma
estrutura correspondente no kernel. Threads nativas do processo também
estão sujeitas a limites do sistema operacional/`cgroup`
(`pids_max`/`ulimit -u`). Por isso esse tipo de vazamento normalmente se
manifesta como `OutOfMemoryError: unable to create native thread` (memória
nativa ou limite de threads esgotados) muito antes de qualquer sintoma
relacionado ao heap.

## Como isso se manifesta na prática

- 1 requisição a `POST /notifications` → 2 threads novas e permanentes
  (`pool-N-thread-1`, `pool-N-thread-2`, com `N` crescente a cada
  requisição), independente do resultado da chamada (sucesso, timeout via
  Resilience4j, ou fallback do circuit breaker — nenhum desses caminhos
  fecha o executor).
- Sob um teste de carga sustentado, o número de threads vivas do
  `notification-service` cresce linearmente com o número de requisições,
  visível em Grafana/Prometheus pela métrica `jvm_threads_live_threads`
  (ou `jvm_threads_peak_threads`) exposta por `/actuator/prometheus`.
- Como o limite de memória do container é de `512Mi`
  (`k8s/notification-service.yaml`) e cada thread reserva memória nativa
  para sua stack, o `OutOfMemoryError: unable to create native thread`
  costuma aparecer bem antes de qualquer limite de heap ser atingido —
  em geral após alguns milhares de requisições, dependendo do `-Xss`
  efetivo e da memória disponível no pod.

## Como confirmar o diagnóstico

0. Rodar `./scripts/reproduce-bug-002.sh` (requer os pods de pé e
   `minikube tunnel` ativo) — o script dispara carga concorrente contra
   `POST /notifications` e imprime `jvm.threads.live` a cada 5s. O valor
   sobe durante o teste e, diferente de um pool saudável, **não recua**
   depois que o tráfego para, o que já evidencia o vazamento.
1. Observar em Grafana/Prometheus o painel de threads do
   `notification-service` (métrica `jvm_threads_live_threads`) durante um
   teste de carga contra `POST /notifications` — a curva sobe de forma
   praticamente linear e nunca recua, mesmo em períodos sem tráfego (threads
   ociosas de pools nunca encerrados continuam vivas).
2. Tirar um thread dump do pod (`jstack <pid>` via `kubectl exec`, ou
   habilitar `management.endpoint.threaddump` e baixar via
   `/actuator/threaddump`) e observar um número muito grande de threads com
   nomes no padrão `pool-<N>-thread-<1|2>`, a maioria em estado `WAITING`
   (esperando por tarefas que nunca chegarão, pois cada pool só recebe 2
   tarefas e depois é abandonado) — o número de pools distintos (`N`) deve
   ser muito próximo do número de requisições já recebidas.
3. Alternativamente, monitorar `ps -o nlwp <pid>` (número de threads leves
   do processo) ou `cat /proc/<pid>/status | grep Threads` dentro do
   container ao longo do teste de carga e observar o mesmo crescimento
   monotônico — o que já comprova que threads estão sendo criadas sem
   nunca serem destruídas.

## Solução

> A correção aplicada neste repositório segue exatamente o raciocínio
> abaixo; a decisão de usar um `ExecutorService` compartilhado (em vez de
> migrar para `WebClient` reativo ou `@Async`) está registrada em
> [`docs/ADR/0002-executor-compartilhado-notification.md`](ADR/0002-executor-compartilhado-notification.md).

Duas correções, uma para cada problema:

1. **Não criar um `ExecutorService` por requisição** — extrair para um
   único pool compartilhado, criado uma vez, como bean gerenciado pelo
   Spring:

   ```java
   @Configuration
   public class NotificationExecutorConfig {

       @Bean(destroyMethod = "shutdown")
       public ExecutorService notificationExecutor() {
           return Executors.newFixedThreadPool(8);
       }
   }
   ```

   O `destroyMethod = "shutdown"` garante que o Spring encerre o pool de
   forma ordenada quando o contexto da aplicação for finalizado (ex.:
   `SIGTERM` do Kubernetes), evitando também vazamento no *shutdown*.

2. **Reaproveitar esse pool em todas as requisições**, em vez de criar um
   novo a cada chamada:

   ```java
   private final ExecutorService notificationExecutor;

   public NotificationController(PaymentClient paymentClient,
                                  ShippingClient shippingClient,
                                  ExecutorService notificationExecutor) {
       this.paymentClient = paymentClient;
       this.shippingClient = shippingClient;
       this.notificationExecutor = notificationExecutor;
   }

   @PostMapping
   public ResponseEntity<Map<String, Object>> createNotification(@RequestBody Map<String, Object> request) {
       String paymentId = (String) request.getOrDefault("paymentId", "UNKNOWN");
       String sku       = (String) request.getOrDefault("sku", "UNKNOWN");

       CompletableFuture<Map<String, Object>> paymentFuture =
               CompletableFuture.supplyAsync(() -> paymentClient.getPayment(paymentId), notificationExecutor);
       CompletableFuture<Map<String, Object>> shippingFuture =
               CompletableFuture.supplyAsync(() -> shippingClient.getShipment(sku), notificationExecutor);

       CompletableFuture.allOf(paymentFuture, shippingFuture).join();

       return ResponseEntity.status(201).body(Map.of(
               "notification", request,
               "paymentStatus", paymentFuture.join(),
               "shippingStatus", shippingFuture.join()
       ));
   }
   ```

   Com um pool de tamanho fixo e reaproveitado, o número de threads do
   processo passa a ter um teto previsível (`8`, no exemplo), independente
   do volume de requisições — a mesma lógica de "limite explícito" aplicada
   ao cache no BUG-001, agora para threads em vez de entradas de mapa.

### Correções complementares (defesa em profundidade)

- Expor o tamanho do pool e a fila de tarefas pendentes como métricas
  Micrometer (`ExecutorServiceMetrics.monitor(...)`) para que o
  comportamento do executor fique visível em Grafana, e alertar sobre
  crescimento anômalo de `jvm_threads_live_threads` antes de chegar a um
  `OutOfMemoryError`.
- Preferir, quando possível, não usar um `ExecutorService` manual para esse
  tipo de paralelismo de I/O e considerar `WebClient` reativo ou
  `CompletableFuture` sobre o executor padrão do Spring
  (`@Async` + `TaskExecutor` configurado), que já é gerenciado pelo ciclo de
  vida do contexto.
- Adicionar um teste de carga automatizado (ex.: k6 ou Gatling) no CI que
  rode um número alto de requisições contra `POST /notifications` e falhe
  se `jvm_threads_live_threads` crescer de forma monotônica sem estabilizar
  — a mesma estratégia já sugerida no BUG-001 para heap, aplicada a
  threads.
